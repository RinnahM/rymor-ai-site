The Problem

Most "AI chat app" portfolio projects are a thin frontend calling someone else's API — that proves you can call an API, not that you understand what happens between a user's question and a grounded answer. I set out to build one that shows the whole path: chunking, embeddings, similarity search, prompt construction, streaming, and the infrastructure tradeoffs of actually shipping it — not just the chat bubble UI.

What I Built

A retrieval-augmented generation (RAG) app: upload a document, ask a question, get an answer grounded in — and citing — your own content. It runs local-first: the original version used Ollama for the LLM, transformers.js for embeddings, and ran fully offline with zero API keys and zero cost. To put it behind a public URL, the LLM call is abstracted behind a single interface that picks Groq's free hosted inference when a key is present and falls back to local Ollama otherwise — same prompt construction, same streaming interface, different backend depending on where it runs. Storage standardized on Postgres (via a free Neon database) everywhere, since a serverless host's filesystem is ephemeral between invocations and SQLite doesn't survive that. At this scale, chunk embeddings are stored in Postgres and scored with in-memory cosine similarity rather than reaching for a dedicated vector database — right-sized for the data volume, with the retrieval logic isolated so swapping in pgvector later is a contained change, not a rewrite.

What Actually Broke (And How I Fixed It)

PDF uploads failed first: Next.js's bundler was pulling a PDF-parsing dependency into the server bundle in a way that broke its internal worker-thread bootstrapping, so the route threw an unhandled exception and the client got an unreadable "Unexpected end of JSON input" instead of a real error. Fixed by marking the PDF libraries as external so Node could `require` them natively, and by wrapping the upload route in a try/catch that always returns a structured JSON error instead of crashing silently.

The second bug only showed up on the real deployment: the first live upload on Vercel came back a 500 with libonnxruntime.so.1.14.0: cannot open shared object file. The local embedding model depends on a native binary that Vercel's file tracer doesn't detect automatically, since it's loaded dynamically based on platform/architecture rather than a static import — locally the full node_modules tree is just present, so nothing catches this until the function actually ships without it. Fixed by explicitly force-including that native binary in the relevant serverless functions. Both bugs trace back to the same lesson: a bundler optimizing for the smallest possible bundle will happily strip a file a native addon needs at runtime, and it fails silently until you hit it in production.
